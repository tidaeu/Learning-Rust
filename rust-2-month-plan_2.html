Phase 1: Core Rust
Week 0 (soft start) + Weeks 1–3 · The language + the borrow checker · reach reading fluency
GOAL: Get the toolchain working, run your first Rust program, and feel the compile → error → fix loop. No timer.
You already program well — Rust's syntax will feel familiar; only ownership is new. Use Week 0 to set up and build the habit, not to learn hard material.
SETUP (1 HR)
Install the toolchain via rustup (rustup.rs) — gives rustc, cargo, and std. Verify with `rustc --version` (you want 1.85+ for the 2024 edition). Run `rustup component add clippy`.
Wire up Zed for Rust: rust-analyzer ships built in. Turn ON inlay type hints and format-on-save — seeing inferred types everywhere is a huge aid while ownership is still new. Your vim keymap carries over untouched.
Create a PUBLIC GitHub repo "rust-sprint" — this is your portfolio repo, not a scratch pad. Clean README: "Learning Rust in 2 months — data-structures, async networking, WASM, and a dev tool."
`cargo new hello-rust && cd hello-rust && cargo run`. Open Cargo.toml — note `edition = "2024"`. This is your project manifest (like package.json/pyproject).
Bookmark the four pillars you'll live in: the Book (Brown interactive edition, rust-book.cs.brown.edu), Rustlings, Rust by Example, and the std docs (doc.rust-lang.org/std).
THURSDAY — FIRST TASTE (1–2 HRS)
📗 LEARN: Book Ch. 1–3 (setup, guessing game, common concepts). The guessing game is a fast tour of match, Result, crates, and shadowing.
🔥 DRILL: `cargo install rustlings`, then `rustlings init`. Do the intro + variables sections. Run `rustlings` and let it guide you exercise by exercise.
FRIDAY — OWNERSHIP, FIRST CONTACT (1–2 HRS)
📗 LEARN: Book Ch. 4 (Ownership) — slowly. Move semantics, borrowing, slices. This is THE chapter and the only thing here that's genuinely new coming from Python. Don't rush it.
🔒 BUILD: First public commit — a tiny CLI that reverses its argument and counts characters, using `std::env::args()`. Push to rust-sprint. The point is to feel rustc's error messages (they're famously good).

THE MAKE-OR-BREAK WEEK: Everything else depends on this, so it's front-loaded. Expect to fight the compiler — that's the learning.
MONDAY — OWNERSHIP DEEP (2.5 HRS)
📗 LEARN: Re-read Book Ch. 4 and take the Brown edition quizzes — they're built to catch the exact misconceptions beginners have. Make move vs copy vs clone crystal clear.
🔥 DRILL: Rustlings `move_semantics` (all exercises).
🔒 BUILD: Start a "Rust" Anki deck. First cards: when does a value move vs copy? what is a borrow? why can't a &mut coexist with a shared &?
TUESDAY — REFERENCES & SLICES (2.5 HRS)
📗 LEARN: Book Ch. 4.2–4.3 (references, slices). Internalize the aliasing rule: one &mut XOR many &. This single rule is why Rust prevents data races.
🔥 DRILL: Rustlings `primitive_types`, `vecs`, `strings` — these force the &str vs String distinction, the #1 early friction point.
🔒 BUILD: Write helpers that take `&str` (not String) and `&[T]` (not Vec<T>) — practice borrowing in APIs instead of taking ownership.
WEDNESDAY — STRUCTS, ENUMS & MATCH (2.5 HRS)
📗 LEARN: Book Ch. 5–6 (structs, methods, enums, match, Option). Coming from Python, `Option<T>` replacing null is the key shift — the compiler forces you to handle "nothing".
🔥 DRILL: Rustlings `structs`, `enums`, `options`.
🔒 BUILD: Model a small domain as a data-carrying enum (e.g. a `Shape` with Circle/Rect variants and an `area()` method dispatched via match).
THURSDAY — LIFETIMES (2.5 HRS)
📗 LEARN: Book Ch. 10.3. Lifetimes are NOT about how long values live — they're labels relating references so the compiler can prove nothing dangles. Then watch Jon Gjengset's "Crust of Rust: Lifetime Annotations" — the best explanation anywhere.
🔥 DRILL: Rustlings `lifetimes`.
🔒 BUILD: Anki — "what does a lifetime annotation actually mean?", "when do you need explicit lifetimes vs elision?"
FRIDAY — CONSOLIDATE (2.5 HRS)
🔥 DRILL: Finish any leftover Rustlings from intro through lifetimes.
🔒 BUILD: Refactor Week 0's CLI to use borrowed slices throughout and add a custom struct. Notice how much less you reach for `.clone()` now.
📗 LEARN: Read "References and Borrowing" in Rust by Example for a second angle on the same ideas.
SATURDAY — PROJECT: TOKENIZER (3 HRS)
Saturday project — build a tokenizer/lexer for arithmetic ("3 + 4 * 2" → tokens). Input: &str. Output: Vec<Token> where Token is an enum (Number(f64), Plus, Star, LParen, …). This forces ownership, borrowing, enums, and char iteration together. Commit with a README.

THE HASHMAP WEEK: Generics + traits land first because you need both to build a real generic hashmap — your requested centerpiece project.
MONDAY — GENERICS (2.5 HRS)
📗 LEARN: Book Ch. 10.1 (generics + monomorphization). Generic functions, structs, enums, and the zero-cost story (generics compile to specialized code).
🔥 DRILL: Rustlings `generics`.
🔒 BUILD: Write a generic function with a trait bound (e.g. `fn largest<T: PartialOrd>(items: &[T]) -> &T`).
TUESDAY — TRAITS (2.5 HRS)
📗 LEARN: Book Ch. 10.2 + Ch. 17. Traits are interfaces. Learn trait bounds, `impl Trait`, and `dyn Trait` (dynamic dispatch) vs generics (static dispatch). Meet the core derives: Debug, Clone, PartialEq, Eq, Hash, Default.
🔥 DRILL: Rustlings `traits`.
🔒 BUILD: Anki — "static vs dynamic dispatch, when each?", "what does #[derive(Hash)] require?"
WEDNESDAY — ERROR HANDLING (2.5 HRS)
📗 LEARN: Book Ch. 9 (Result, the `?` operator, panic vs recoverable). Then the ecosystem standard: `thiserror` for libraries (derive your error enum) and `anyhow` for applications (easy propagation). This is how real Rust handles errors — no exceptions.
🔥 DRILL: Rustlings `error_handling`.
🔒 BUILD: Give your Week 1 tokenizer real error handling — return a `TokenizeError` (via thiserror) instead of panicking on bad input.
THURSDAY — HASHMAP FROM SCRATCH, PART 1 (2.5 HRS)
📗 LEARN: Refresh hashing + separate chaining vs open addressing (you've built one in Python — now do it in Rust). Skim the real std::collections::HashMap docs to see the API you're emulating.
🔒 BUILD: Start `myhashmap`. Define `struct HashMap<K, V> { buckets: Vec<Vec<(K, V)>>, len: usize }`. Implement `new()` and `insert(K, V)` with `K: Hash + Eq`, using std's DefaultHasher to map keys to a bucket index.
FRIDAY — HASHMAP FROM SCRATCH, PART 2 (2.5 HRS)
🔒 BUILD: Implement `get(&self, key: &K) -> Option<&V>`, `remove`, `contains_key`, and the interesting part — resize/rehash when the load factor passes ~0.75.
🔥 DRILL: Write `#[test]` unit tests covering insert/get/remove/resize and hash collisions. Run `cargo test`.
SATURDAY — PROJECT: FINISH & POLISH HASHMAP (3–4 HRS)
Saturday project — make it ergonomic: implement the `Index` trait so `map[key]` works, add an `iter()` / `IntoIterator`, and benchmark against std's HashMap on 1M inserts (criterion or simple timing). README explaining chaining vs open addressing, your load factor, and the numbers. This is a genuine portfolio centerpiece.

READING-FLUENCY WEEK: Iterators + std combinators are how idiomatic Rust is written, so this is what makes you able to READ any Rust codebase.
MONDAY — CLOSURES & ITERATORS (2.5 HRS)
📗 LEARN: Book Ch. 13. Fn/FnMut/FnOnce, the Iterator trait, map/filter/fold/collect. Iterators are lazy and `collect()` is type-driven — that part is new vs Python comprehensions, but it'll click fast.
🔥 DRILL: Rustlings `iterators` (all — they're meaty and worth it).
🔒 BUILD: Rewrite a loop-heavy function from an earlier project as an iterator chain.
TUESDAY — SMART POINTERS (2.5 HRS)
📗 LEARN: Book Ch. 15 (Box, Rc, RefCell, Deref, Drop). When you need heap allocation (Box), shared ownership (Rc), or interior mutability (RefCell) — and why `Rc<RefCell<T>>` exists.
🔥 DRILL: Rustlings `smart_pointers`, `box`.
🔒 BUILD: Anki — "Box vs Rc vs RefCell, what does each give you?", "what is interior mutability?"
WEDNESDAY — MODULES, CRATES, WORKSPACES (2.5 HRS)
📗 LEARN: Book Ch. 7 (modules, paths, pub, use) + Ch. 14 (Cargo, workspaces, publishing). How to lay out a real multi-file project.
🔒 BUILD: Reorganize your hashmap into a proper library crate with modules, add doc comments (`///`), and run `cargo doc --open`.
THURSDAY — STD FLUENCY + LEETCODE IN RUST (2.5 HRS)
📗 LEARN: Tour std::collections (Vec, HashMap, BTreeMap, HashSet, VecDeque) and the Option/Result combinators (map, and_then, ok_or, unwrap_or_else) — they appear in nearly every file.
🔥 DRILL: Solve 2–3 LeetCode mediums in Rust — you know the patterns from your Python prep, now feel them in Rust (Two Sum → HashMap, Group Anagrams → entry API, Valid Parentheses → Vec as stack).
FRIDAY — TESTING, CLIPPY, FMT, CI (2.5 HRS)
📗 LEARN: Book Ch. 11 (unit tests, integration tests in tests/, #[cfg(test)]). Then run `cargo clippy` on everything and fix every lint — clippy is the best Rust teacher you have. `cargo fmt` to finish.
🔒 BUILD: Add a GitHub Actions CI that runs `cargo test`, `cargo clippy -- -D warnings`, and `cargo fmt --check`. A green CI badge on your repo signals professionalism.
SATURDAY — PROJECT: rgrep CLI (3–4 HRS)
Saturday project — build a grep clone (`rgrep`): read a file, find a pattern, print matching lines with line numbers and optional case-insensitivity. Use `clap` for args (the standard CLI crate), Result-based errors, and iterators for line processing. Bonus: colored output (`colored` crate) and `--recursive`. The canonical first "real tool".

Phase 2: Async & Networking
Weeks 4–5 · Concurrency, async/await, and real network/backend services · your networking + cloud interest
THE BRIDGE TO NETWORKING: First fearless concurrency (threads), then the async model that every Rust network service is built on.
MONDAY — THREADS & SHARED STATE (2.5 HRS)
📗 LEARN: Book Ch. 16. Threads, `move` closures, channels (mpsc), and Arc + Mutex for shared state. The Send and Sync marker traits are WHY the borrow checker turns data races into compile errors.
🔥 DRILL: Rustlings `threads`.
🔒 BUILD: Spawn N threads that each sum part of a large Vec and combine the results over a channel.
TUESDAY — ATOMICS & THE WHY (2.5 HRS)
📗 LEARN: Read Ch. 1–2 of Mara Bos's "Rust Atomics and Locks" (free at marabos.nl/atomics) — the clearest account of Send/Sync, Arc, and Mutex internals. You don't need all of it; Ch 1–2 cement the model.
🔒 BUILD: Anki — "what do Send and Sync mean?", "Arc vs Rc, why two types?", "why are data races a compile error in Rust?"
WEDNESDAY — ASYNC CONCEPTS (2.5 HRS)
📗 LEARN: Book Ch. 17 (async/await) + the async book intro. The model: an `async fn` returns a Future, Futures are LAZY (do nothing until polled), and you need a runtime to drive them. Concurrency without (necessarily) parallelism.
🔥 DRILL: Watch Jon Gjengset "Crust of Rust: async/await" for the underlying mental model (poll, wakers, the state machine).
THURSDAY — TOKIO (2.5 HRS)
📗 LEARN: Work through the official tokio tutorial (tokio.rs/tokio/tutorial) — Hello Tokio, Spawning, Shared State, Channels. tokio is THE async runtime (async-std is deprecated). Learn #[tokio::main], tokio::spawn, .await, tokio::select!.
🔒 BUILD: Fetch 5 URLs concurrently with `reqwest` and report which returns first (tokio::join! or spawn + collect).
FRIDAY — ASYNC PATTERNS (2.5 HRS)
📗 LEARN: Async channels (tokio::sync::mpsc), tokio::sync::Mutex, and timeouts (tokio::time::timeout). Key gotcha: holding a std Mutex across an `.await` is a bug — learn why.
🔒 BUILD: Anki — "why are futures lazy?", "std Mutex vs tokio Mutex, when each?", "what does `.await` actually do?"
SATURDAY — PROJECT: ASYNC TCP CHAT SERVER (3–4 HRS)
Saturday project — build an async TCP server with tokio. Start with an echo server (TcpListener, accept loop, one task per connection). Then extend to a multi-client chat: broadcast each message to all clients via tokio::sync::broadcast. Test with several `nc`/telnet clients. Real network programming, strong portfolio piece.

THE BACKEND WEEK: Build a real HTTP service end to end — routing → JSON → state → database → deployed. This is the cloud/backend story.
MONDAY — HTTP & AXUM BASICS (2.5 HRS)
📗 LEARN: Intro to axum — the standard modern web framework (built on tokio + hyper + tower; axum 0.8 dropped the async_trait requirement). Routers, handlers, and extractors (Path, Query, Json, State). Read the axum docs + examples.
🔒 BUILD: A "hello axum" server with GET /health, GET /hello/:name, and a route returning JSON.
TUESDAY — JSON & SERDE (2.5 HRS)
📗 LEARN: serde + serde_json — THE serialization framework, used everywhere. `#[derive(Serialize, Deserialize)]`. See how axum's `Json<T>` extractor uses it.
🔒 BUILD: A CRUD JSON API for an in-memory resource (notes/todos) behind `Arc<RwLock<…>>` — POST/GET/PUT/DELETE with correct status codes and JSON error bodies.
WEDNESDAY — STATE, MIDDLEWARE, OBSERVABILITY (2.5 HRS)
📗 LEARN: Shared state (State extractor), middleware via tower (logging, timeouts), and error handling by implementing `IntoResponse` for your error type. Add structured logging with `tracing` + `tracing-subscriber` (the standard observability stack).
🔒 BUILD: Add request logging (tracing) and a global error handler returning consistent JSON error envelopes.
THURSDAY — DATABASES WITH SQLX (2.5 HRS)
📗 LEARN: sqlx — async, compile-time-checked SQL (no ORM magic). Connect to SQLite or Postgres, run migrations, and query with `query!` / `query_as!` macros that verify your SQL against the schema at compile time.
🔒 BUILD: Swap your API's in-memory store for a real SQLite database via sqlx, with migrations. Now you have a persistent backend.
FRIDAY — LOWER-LEVEL NETWORKING (2.5 HRS)
📗 LEARN: Drop a layer — read about `hyper` (the HTTP library under axum) and the TCP → HTTP → framework stack. Skim Book Ch. 21 (the multithreaded web server) to contrast a hand-rolled thread-per-connection server with the async approach.
🔒 BUILD: Anki — "what does an extractor do in axum?", "why is sqlx compile-time checked?", "TCP vs HTTP, what layer is each?"
SATURDAY — PROJECT: DEPLOYABLE BACKEND (3–4 HRS)
Saturday project — polish your axum + sqlx service into something deployable: config via env vars, a multi-stage Dockerfile (→ tiny image), and a health endpoint. Deploy free (Shuttle.rs, Fly.io, or Railway) so you have a LIVE URL. README with the architecture + link. Exactly the project that lands Rust backend interviews.

Phase 3: WASM & Developer Tooling
Weeks 6–8 · WebAssembly (browser + server) and dev tooling, then a capstone · your strongest interests
YOUR TOP INTEREST: Get Rust compiling to WASM and running in a browser, then prove a real speedup over JS.
MONDAY — WASM FUNDAMENTALS (2.5 HRS)
📗 LEARN: What WASM is and why Rust is a top choice (no GC → tiny, fast binaries). Read the "Rust and WebAssembly" book intro. Install: `rustup target add wasm32-unknown-unknown`, then `wasm-pack`.
🔒 BUILD: Hello WASM — `cargo new --lib`, set `crate-type = ["cdylib"]`, export `add(a,b)` and `greet() -> String` with `#[wasm_bindgen]`, build with `wasm-pack build --target web`, and call it from a tiny index.html. Seeing Rust run in your browser is the unlock.
TUESDAY — WASM-BINDGEN DEEPER (2.5 HRS)
📗 LEARN: The wasm-bindgen guide. Passing strings, structs, and closures across the JS↔WASM boundary, and why that boundary is expensive (data copying, UTF-8↔UTF-16) — so prefer few, coarse-grained calls. Meet js-sys (JS builtins) and web-sys (DOM / Web APIs).
🔥 DRILL: Export a Rust struct (e.g. a `Counter` with methods) and drive it from JS; pass a JS array in and return a transformed one.
WEDNESDAY — THE GAME OF LIFE (2.5 HRS)
📗 LEARN + 🔒 BUILD: Work through the canonical Conway's Game of Life tutorial in the Rust+WASM book — the best end-to-end intro. Rust owns the universe state; JS renders to canvas. You'll use web-sys and handle the boundary properly.
THURSDAY — PERFORMANCE & SIZE (2.5 HRS)
📗 LEARN: Benchmark a compute-heavy function (mandelbrot, image filter, your parser) in Rust-WASM vs plain JS. Shrink the binary: `wasm-opt`, `opt-level = "z"`, `lto = true`, strip, and inspect with `twiggy`. Add `console_error_panic_hook` for debugging.
🔒 BUILD: Take one genuinely compute-heavy task and prove the WASM speedup with real numbers in your README.
FRIDAY — FRONTEND INTEGRATION (2.5 HRS)
📗 LEARN: How WASM fits real apps — `trunk` (the Rust-WASM web bundler) for all-Rust frontends, or publishing a wasm-pack package to npm for a JS/TS app. Glance at Leptos/Yew/Dioxus so you know the landscape — but don't fall down that rabbit hole now.
🔒 BUILD: Anki — "why is the JS↔WASM boundary expensive?", "when is Rust-WASM worth it vs JS?", "what does wasm-bindgen generate?"
SATURDAY — PROJECT: WASM IMAGE FILTER (3–4 HRS)
Saturday project — an in-browser image filter app. Rust-WASM does pixel processing (grayscale, blur, edge detection); JS handles upload + canvas. Clean API, optimized binary, and a before/after with a JS-vs-WASM timing comparison. Deploy to GitHub Pages. A flagship piece showcasing your top interest.

WASM BEYOND THE BROWSER + DEV TOOLING: Where WASM meets your cloud interest (edge, plugins), then the macro machinery behind every dev tool.
MONDAY — WASI & WASMTIME (2.5 HRS)
📗 LEARN: WASM outside the browser. WASI gives sandboxed access to files/env/sockets. Targets were renamed: `wasm32-wasip1` (Tier 2, stable) and `wasm32-wasip2` (adds networking/sockets). Install Wasmtime, the standalone runtime.
🔒 BUILD: Compile a Rust CLI to `wasm32-wasip1` and run it with `wasmtime`. Feel the "compile once, run sandboxed anywhere" model.
TUESDAY — THE COMPONENT MODEL (2.5 HRS)
📗 LEARN: The WebAssembly Component Model + WIT (Wasm Interface Types) — portable, language- agnostic components. `cargo component` + `wit-bindgen` define interfaces in .wit and generate bindings. Cutting-edge and a genuine CV differentiator.
🔒 BUILD: Define a small WIT interface, implement it in Rust with `cargo component`, and inspect / run it with `wasm-tools` / Wasmtime.
WEDNESDAY — WHERE SERVER WASM MATTERS (2.5 HRS)
📗 LEARN: Real-world server WASM — edge compute (Fastly, Cloudflare Workers), safe plugin systems (run untrusted user code via wasmtime), and FaaS. The why: instant cold starts, strong sandboxing, language-agnostic. This connects WASM to your cloud/backend interest.
🔒 BUILD: Anki — "WASI p1 vs p2?", "what problem does the Component Model solve?", "why run WASM server-side instead of containers?"
THURSDAY — MACROS PART 1: DECLARATIVE (2.5 HRS)
📗 LEARN: Start dev tooling. `macro_rules!` (declarative macros) — pattern-matching code generation. Book Ch. 20 macro section + skim "The Little Book of Rust Macros". When macros earn their keep vs functions/generics.
🔒 BUILD: Write a useful `macro_rules!` macro — e.g. a `hashmap!{ "a" => 1 }` literal, or a test-case generator.
FRIDAY — MACROS PART 2: PROCEDURAL (2.5 HRS)
📗 LEARN: Procedural macros — the machinery behind `#[derive(...)]` and serde. The `proc-macro2`, `syn`, and `quote` crates. Watch Jon Gjengset's "Crust of Rust: Declarative & Procedural Macros". Deep dev-tooling territory.
🔒 BUILD: Write a simple derive macro (e.g. `#[derive(Builder)]` generating a builder). Even a small one teaches you how the whole derive ecosystem works.
SATURDAY — PROJECT: PICK YOUR TRACK (3–4 HRS)
Saturday project (pick one) — (a) a WASM plugin host: a Rust app using wasmtime that loads and runs a user-supplied .wasm plugin against a defined WIT interface (sandboxed extensibility — very impressive); OR (b) a working derive-macro crate published to crates.io with docs and tests. Either is a serious WASM / dev-tooling piece.

TIE IT TOGETHER: The advanced corners that let you READ any Rust code (unsafe, FFI), plus a capstone that plays to your three interests.
MONDAY — UNSAFE & THE RUSTONOMICON (2.5 HRS)
📗 LEARN: `unsafe` Rust — raw pointers, unsafe fn, the five superpowers, and the contract YOU must uphold. Read the Rustonomicon intro chapters. Key insight: unsafe doesn't disable the borrow checker; it grants 5 specific abilities while you guarantee soundness. Essential for reading std/library internals.
🔒 BUILD: Reimplement a tiny thing with unsafe correctly (e.g. `split_at_mut` by hand, or a simple bump allocator) and document the safety invariants.
TUESDAY — FFI (2.5 HRS)
📗 LEARN: Foreign Function Interface — calling C from Rust (`extern "C"`, `#[link]`) and exposing Rust to C (`#[no_mangle]`, now requiring `unsafe` in the 2024 edition). `bindgen` / `cbindgen`. FFI also underpins Python extensions — relevant to you via PyO3.
🔒 BUILD: Call a simple C function from Rust, OR expose a Rust function to Python with PyO3 and call it from a Python script.
WEDNESDAY — PERFORMANCE & PROFILING (2.5 HRS)
📗 LEARN: Make Rust fast — criterion (statistical benchmarks), `cargo flamegraph` (profiling), avoiding needless allocations/clones, &str over String, iterators over manual loops, and release vs debug builds. Read a couple of fasterthanlime's performance posts.
🔒 BUILD: Profile and optimize an earlier project (the hashmap or image filter). Show before/after numbers.
THURSDAY — READ REAL CODE + IDIOMS (2.5 HRS)
📗 LEARN: Reading-fluency drill — read the source of a small, well-written crate (itertools internals, or a module of ripgrep/fd). Skim "Effective Rust" idioms and "Rust for Rustaceans" chapters on API design + type layout. Goal: open ANY Rust file and follow it.
🔒 BUILD: Find a "good first issue" on a Rust project you've used (clap, axum, a tokio crate). Reading the issue + codebase counts; a small PR is a bonus and a fantastic signal.
FRIDAY — CAPSTONE PLANNING + POLISH (2.5 HRS)
🔒 BUILD: Pick your capstone (see Saturday) and scaffold it today — structure, CI, README skeleton.
📗 LEARN: Skim your Anki deck end to end. Walk the Book's table of contents and confirm no chapter is still a black box — anything fuzzy, mark it Shaky.
SATURDAY — CAPSTONE: SHIP IT (4 HRS)
Saturday capstone (pick what excites you most) — 1) a real developer CLI (fast file finder, log parser, code-stats, scaffolder) polished with clap, color, config, tests, CI, published to crates.io; 2) a networked service with a WASM twist — your axum backend running user-submitted WASM plugins via wasmtime (networking + WASM + backend in one, the strongest signal); or 3) a browser WASM app (extend the image filter, or a WASM data-viz / mini-game) deployed live.
